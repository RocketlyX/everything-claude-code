# 规划命令

此命令调用 **planner** 代理在编写任何代码前创建全面的实施计划。

## 此命令的功能

1. **重述需求** - 澄清需要构建的内容
2. **识别风险** - 揭示潜在问题和阻碍
3. **创建步骤计划** - 将实施分解为阶段
4. **等待确认** - 必须在继续前获得用户批准

## 何时使用

在以下情况使用 `/plan`：
- 开始新功能
- 进行重大架构更改
- 处理复杂重构
- 多个文件/组件将受影响
- 需求不清楚或模糊

## 工作原理

planner 代理将：

1. **分析请求**并用清晰的术语重述需求
2. **分解为阶段**包含具体、可操作的步骤
3. **识别依赖关系**组件之间
4. **评估风险**和潜在阻碍
5. **估计复杂度**（高/中/低）
6. **展示计划**并等待你的明确确认

## 使用示例

```
用户：/plan 我需要在市场结算时添加实时通知

代理（planner）：
# 实施计划：实时市场结算通知

## 需求重述
- 当用户关注的市场结算时向他们发送通知
- 支持多种通知渠道（应用内、邮件、webhook）
- 确保通知可靠送达
- 包含市场结果和用户仓位结果

## 实施阶段

### 阶段 1：数据库模式
- 添加 notifications 表，列包括：id, user_id, market_id, type, status, created_at
- 添加 user_notification_preferences 表用于渠道偏好
- 在 user_id 和 market_id 上创建索引以提高性能

### 阶段 2：通知服务
- 在 lib/notifications.ts 中创建通知服务
- 使用 BullMQ/Redis 实现通知队列
- 为失败投递添加重试逻辑
- 创建通知模板

### 阶段 3：集成点
- 接入市场结算逻辑（当状态变为 "resolved" 时）
- 查询市场中所有有仓位的用户
- 为每个用户入队通知

### 阶段 4：前端组件
- 在头部创建 NotificationBell 组件
- 添加 NotificationList 模态框
- 通过 Supabase 订阅实现实时更新
- 添加通知偏好页面

## 依赖
- Redis（用于队列）
- 邮件服务（SendGrid/Resend）
- Supabase 实时订阅

## 风险
- 高：邮件投递率（需要 SPF/DKIM）
- 中：每市场 1000+ 用户时的性能
- 中：市场频繁结算时的通知垃圾
- 低：实时订阅开销

## 估计复杂度：中
- 后端：4-6 小时
- 前端：3-4 小时
- 测试：2-3 小时
- 总计：9-13 小时

**等待确认**：是否继续此计划？（是/否/修改）
```

## 重要说明

**关键**：planner 代理**不会**编写任何代码，直到你用 "是"、"继续" 或类似肯定回复明确确认计划。

如果你想要更改，回复：
- "修改：[你的更改]"
- "不同方法：[替代方案]"
- "跳过阶段 2，先做阶段 3"

## 与其他命令集成

规划后：
- 使用 `/tdd` 以测试驱动开发实现
- 使用 `/build-and-fix` 如果发生构建错误
- 使用 `/code-review` 审查完成的实现
